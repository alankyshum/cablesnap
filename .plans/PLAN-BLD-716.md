# Feature Plan: Wear OS Companion App

## Research Source

- **Pain point observed:** Owner wants to track workouts on-watch without phone-in-hand; no existing free open-source Wear OS workout tracker integrates with a self-hosted/offline-first phone app
- **Frequency:** Owner-prioritized; mirrors recurring Reddit theme ("I switched from Strong/JEFIT because their watch app sucks / costs money / requires their cloud")

## Problem Statement

CableSnap is currently phone-only. For the gym setup the owner uses (cable + bodyweight, single-machine flows), holding a phone between sets is awkward — gym towels, hand chalk, and machine handles fight for the same hand. Strong charges for its Apple Watch app and has no Wear OS app at all; JEFIT requires a paid plan and cloud sync. CableSnap's offline-first, open-source, locally-stored architecture makes a Wear OS companion a unique competitive moat: **the only open-source Wear OS workout tracker that doesn't phone-home anywhere.**

## User Stories

- As a CableSnap user mid-set, I want to log reps/weight from my watch so I can keep my phone in my bag.
- As a CableSnap user mid-rest, I want a wrist-haptic rest-timer that auto-progresses so I don't keep checking my phone screen.
- As a CableSnap user starting a workout, I want to launch a saved template directly from the watch so I never need the phone in the gym.
- As a privacy-conscious user, I want phone↔watch data to never leave my devices — no Google account sync, no cloud server.

## Proposed Solution

### Overview

Native Kotlin + Jetpack Compose for Wear OS app paired to the CableSnap phone app via a custom Expo native module that bridges Google's Wearable Data Layer API. Watch app reads templates and writes workout/set events; phone app remains the source of truth (SQLite). All sync is point-to-point Bluetooth via Google's `MessageClient` / `DataClient` — no cloud dependency.

### UX Design

**Watch screens** (per owner spec, with refinements):

1. **Template Library** — `ScalingLazyColumn` listing synced templates. Tap → start workout immediately (no confirm). Empty state: "No templates synced — open CableSnap on your phone."
2. **Live Workout Tracking** — Per-exercise card. Horizontal swipe between exercises (no swipe past last). Header: exercise name (truncated, marquee on focus). Body: weight stepper (±, large tap targets, rotary input), reps stepper (±, rotary), set counter ("Set 2 of 4"). Footer: "Complete Set" pill (full-width).
3. **Rest Timer** — Circular `ProgressRing` countdown of the user's configured rest duration. Buttons: "+30s" (left), "Skip" (ghost, right). Haptic double-buzz on completion. Auto-returns to Live Tracking after the buzz.
4. **Set Management** (long-press on Set Counter) — Add Set / Delete Current Set (danger style on Delete).
5. **Workout Controls** (long-press on header) — Active: Pause / Finish. Paused: Resume / Cancel (long-press confirms cancel).

**A11y**:
- All steppers expose `contentDescription` reading current value + units ("forty-five kilograms").
- Rotary input is the primary input modality — touch is fallback.
- Color/contrast: WCAG AA on the gray-on-black Wear theme (must verify on real watch).
- Haptics are mandatory feedback; do not replace with sound only (gym noise).

**Error & empty states**:
- Phone disconnected → live tracking card shows "Phone disconnected — set will sync when reconnected" banner. Set events queue locally until reconnect.
- No templates synced → Template Library shows the empty state above.
- Battery <10% → no app-level intervention (Wear OS handles).

### Technical Approach

#### Repo integration strategy (decided per TL-1)

CableSnap is a **managed Expo** app — `android/` is generated by `expo prebuild`, not checked in. **Chosen path: TL-1(b) — Custom Expo Config Plugin that injects the wear module during prebuild.** Two cooperating in-tree artifacts (per TL-6):

1. **`modules/expo-wearos-bridge/`** — Expo app-local native module (Expo Modules convention). Holds the Kotlin source for the **phone-side bridge** that talks to `MessageClient`/`DataClient`, plus the JS API surface.
2. **`plugins/with-wearos-module.js`** — Expo Config Plugin run during `expo prebuild`. It registers the `:wear` Gradle subproject, copies the templated `wear/` Gradle module from `modules/expo-wearos-bridge/wear-template/` into the prebuild output, and is **flavor-aware** (no-op under `fdroidRelease` — see F-Droid split below).

Fallbacks rejected: (a) eject to bare workflow (one-way migration), (c) sibling repo (loses single-PR review story).

#### F-Droid + Play split (decided per TL-2)

F-Droid's Inclusion Criteria reject any artifact depending on `com.google.android.gms:*`. Mechanically:

- Add a Gradle **product-flavor split** at the phone-app level:
  - `playRelease` — depends on `play-services-wearable`; bridge native module compiled in.
  - `fdroidRelease` — zero GMS deps; bridge module **not compiled in** (Gradle source-set guard).
- Update `fdroid/metadata/com.persoack.cablesnap.yml` `Builds:` section to pin `gradle: [fdroid]`. Update `.github/workflows/scheduled-release.yml` to publish two phone APKs (`playRelease`, `fdroidRelease`) plus the watch APK from `playRelease` only.
- **The watch APK is Play-only.** F-Droid does not host the watch APK. Release notes will state this explicitly.

#### Watch APK distribution (decided per TL-3)

Wear OS 4 dropped embedded watch APKs — the watch APK is a **separate, standalone artifact** the user installs on the watch (not bundled in the phone APK at runtime).

| Aspect | Decision |
|---|---|
| `applicationId` (watch) | `com.persoack.cablesnap.wear` (distinct from phone `com.persoack.cablesnap`) |
| Distribution channel | Google Play Console — same listing as the phone app, "Wear OS" form factor selected. |
| F-Droid hosting | **No** — GMS dep precludes it. |
| Signing | Play App Signing for both phone (`playRelease`) and watch artifacts. F-Droid signing applies only to the GMS-free `fdroidRelease` phone APK. |
| Min Wear OS | Wear OS 4 (Compose-for-Wear 1.4+ baseline) |
| Acquisition path on watch | User installs phone app from Play; the on-watch Play Store surfaces the companion install (standard Wear OS UX). M0/M1 manual QA verifies this before AC1 can be checked. |

#### Phone-side bridge concurrency model (decided per TL-4)

Naive approach (calling `addSet` from a `WearableListenerService` callback) **deadlocks or drops events**. The bridge must respect the RN JS-thread boundary.

```
[Watch UI taps "Complete Set"]
    → DataLayerClient.kt sends MessageClient '/set-complete' (JSON, <1KB)

[Phone, native — background WearableListenerService thread]
  WearMessageReceiverService.kt:
    1. JSON-decode payload
    2. Acquire native bounded queue lock (capacity 200; drop-oldest on overflow with Sentry breadcrumb — Sentry currently unconditionally enabled, see Risk row 1)
    3. If JS runtime loaded → DeviceEventEmitter.emit('cablesnap.watch.setComplete', payload)
       Else → enqueue payload; on next JS init, replay in FIFO order.
    4. Release lock.

[Phone, JS thread]
  hooks/useWatchSync(sessionId):
    Singleton subscriber, mounted from app/_layout.tsx whenever a session is active.
    On 'cablesnap.watch.setComplete' →
      dedupe by (sessionId, exerciseId, setNumber, completedAt) against in-memory LRU (size 64)
      call existing addSet/updateSet/deleteSet from lib/db/session-sets.ts on JS thread
      ack back to native (which dequeues / updates a "consumed" cursor)
```

**Concurrency invariants:** native queue is a bounded `ArrayDeque<ByteArray>` guarded by a single `Mutex`; JS subscriber is mounted exactly once per app lifetime; cold-start scenario covered: a watch event arriving when the JS bundle is not loaded is queued natively until the bundle initializes.

#### Clock authority (decided per QD-6)

- The **watch's monotonic clock** generates `completedAt` for every Complete Set event.
- On reconcile, the phone treats incoming `completedAt` as authoritative for ordering, but **clamps** the value to `[sessionStart, now() + 60s]` to defend against absurd watch-clock drift.
- Concurrent edits on the same `(sessionId, exerciseId, setNumber)` resolve **phone-wins (LWW anchored to phone wall-clock at reconcile time)**.
- A watch event whose `completedAt` clamps materially (>60s shift) emits a one-line Sentry breadcrumb (Sentry currently unconditionally enabled — see Risk row 1).

#### Watch-side durable event queue (decided per QD-7)

`SetEventQueue.kt` uses **Wear OS DataStore (Proto)**, not in-memory state. Capacity 200 events, drop-oldest on overflow with a watch-side notification. On watch app launch, the queue is read and any pending events are replayed via `MessageClient` immediately if a phone connection exists.

#### Multi-watch pairing (decided per QD-8)

Wear OS supports multiple paired nodes. The phone's `WearMessageReceiverService` accepts events from **all** paired nodes but applies **first-event-wins per `(sessionId, exerciseId, setNumber)`** via the dedup LRU. A second event for the same set from a different node is silently dropped with a Sentry breadcrumb. One companion watch per session is the *intended* UX; this guard ensures correctness if the user has multiple paired wearables.

#### Communication protocol

| Direction | Mechanism | Payload | Frequency |
|---|---|---|---|
| Phone → Watch | `DataClient.putDataItem('/templates')` | Full templates list (small JSON) | On template change; on watch reconnect |
| Phone → Watch | `MessageClient.sendMessage('/active-workout')` | Active workout snapshot | On workout start |
| Watch → Phone | `MessageClient.sendMessage('/set-complete')` | { sessionId, exerciseId, setNumber, weight, reps, completedAt } | Every Complete Set |
| Watch → Phone | `MessageClient.sendMessage('/set-mutate')` | Add/delete/update events | As triggered |
| Watch → Phone | `MessageClient.sendMessage('/workout-control')` | pause/resume/finish/cancel | As triggered |

All payloads are JSON-stringified and < 1KB. Phone is single-source-of-truth: the watch optimistically updates UI but reconciles to the phone's response on the next sync tick.

#### Data model touch-points

- No new SQLite tables. All set events flow through the existing `addSet`/`updateSet`/`deleteSet` APIs (`hooks/useSessionActions.ts`, `lib/db/session-sets.ts`).
- New Expo module exposes a `useWatchSync(sessionId)` React hook that listens for watch events and calls the existing handlers.
- Reverse direction: when the phone updates a set (e.g., user edits on phone mid-workout), broadcast a `/set-update` message so the watch reflects.

#### Dependencies

- **Phone (existing stack):** Expo 55, RN 0.83. New custom Expo native module — no third-party RN libraries.
- **Phone (new gradle deps):** `com.google.android.gms:play-services-wearable`.
- **Watch (new gradle module):** Kotlin 1.9+, Compose for Wear OS 1.4+, `play-services-wearable`, AndroidX Wear `ProgressIndicator`/`ScalingLazyColumn`.
- **No JS deps added.**

#### Performance

- Watch app target: 60fps on Pixel Watch 2 / Galaxy Watch 6 (the most common Wear OS 4 devices). Must remain readable on 1.4" display @ 384px.
- Battery: rest-timer screen must not exceed 5% drain per 30-min workout. Use `AlwaysOnDisplay` ambient-mode rendering for the timer.
- Sync latency: < 500ms p99 phone↔watch round-trip in Bluetooth range.

#### Testing strategy

- **Phone module:** unit-test the JS wrapper (jest mocking the native module); integration-test against a fake DataLayer in CI.
- **Watch app:** Compose UI tests via `androidx.compose.ui.test`. Run on the Wear OS 4 emulator in CI.
- **Pairing:** manual on-device QA test plan covering: pair, unpair, phone offline, watch offline, mid-workout disconnect, battery <10%, template change during active workout.

## Review Feedback

### Quality Director (UX) — REQUEST CHANGES (2026-04-27)

**Verdict:** Plan is structurally sound but has **5 blockers** that must be resolved before implementation issues are created. UX surface and milestone sequencing are good; testability and SKILL-alignment claims are over-stated.

**Blockers (must fix in plan):**

1. **AC7 verification method is wrong.** Mitmproxy only inspects TCP/HTTP. Phone↔watch sync runs over Bluetooth via `MessageClient`/`DataClient` and never touches the IP stack — mitmproxy will trivially show "no traffic" even if the implementation were silently phoning home from a different code path. Replace AC7 method with: (a) airplane mode on phone + Bluetooth on, exercise the full feature, all watch→phone events still reconcile; AND (b) `tcpdump` / Network Inspector filtered to the app UID for 30 minutes of typical use shows zero packets to non-LAN destinations. Cite both methods in the AC.

2. **F-Droid + GMS contradiction is unaddressed.** `play-services-wearable` requires Google Play Services on **both** the phone and the watch. F-Droid builds run on devices that may not have GMS (microG, GrapheneOS without sandboxed Google Play, etc.). The plan says F-Droid build is supported and frames this feature as the "open-source moat," but per the dependency table the watch bridge cannot function without GMS. **Blockers (must fix in plan before M1 issue is created):**

**TL-1. Managed-Expo vs. checked-in `android/wear/` is unresolved.** The plan's "Watch app structure" section shows `wear/   (separate Gradle module under android/)` — but this repo has **no checked-in `android/` directory**. CableSnap is a managed Expo app: `android/` is generated by `expo prebuild` on each build (verified: `ls android/` → not present; `expo`/`react-native` versions in package.json; `app.config.ts` is the only Android config surface). This forecloses the structure as drawn. Pick one and document the chosen path with its tradeoffs:

  - **(a) Eject to bare workflow** — check in `android/`, manually merge upstream Expo template changes on every Expo SDK upgrade. One-way migration. Highest ongoing maintenance cost; biggest blast radius (every Expo upgrade now touches Android files). Not recommended.
  - **(b) Custom Expo Config Plugin that injects the wear module during prebuild** — author a plugin (similar in spirit to `plugins/with-release-signing.js`) that adds the `wear/` Gradle module + `settings.gradle` `include(':wear')` line to the prebuild output. **TL-2. F-Droid + GMS contradiction is structural, not a release-note footnote.** F-Droid's build server **rejects** apps depending on `com.google.android.gms:*` libraries — this is a hard policy in the Inclusion Criteria, not a convention. Per QD's blocker #2, both can't be true; here is what's required mechanically:

  - Add a Gradle **product flavor split**: `playRelease` (with `play-services-wearable`, watch bridge enabled) and `fdroidRelease` (no GMS deps, watch bridge module *not compiled in*). The Expo Config Plugin from TL-1(b) must be flavor-aware.

**TL-3. Watch APK distribution is unspecified.** The Wear OS APK is a *separate package* the user installs on the watch — it is NOT bundled inside the phone APK at runtime (Wear OS 4 dropped embedded watch APKs entirely; standalone-only since Wear OS 3). The plan does not specify:
  - Where the watch APK is published (Play Store separately? Same Play Console listing as a Wear-targeted artifact? F-Droid?).
  - Whether F-Droid will host the watch APK at all (almost certainly not, given GMS dep — see TL-2).

  Add a "Distribution" section to Technical Approach answering these. Without it, AC1 (Pairing) is untestable end-to-end because we don't know how the user gets the watch APK onto the watch.

**TL-4. Phone-side message handling is in a JS-thread danger zone.** The plan says watch events route through `addSet`/`updateSet`/`deleteSet` (verified at `lib/db/session-sets.ts:119`, exported from `lib/db/index.ts`). Those are async functions called from React hooks (`hooks/useSessionActions.ts`). On Android, `MessageClient` callbacks fire on a background `WearableListenerService` thread — calling into RN's JS runtime from there requires going through `DeviceEventEmitter` / `RCTEventEmitter`, not a direct function call. Naive direct calls deadlock or drop events.

  Specify the bridge pattern explicitly:
  - `WearableListenerService` (Java/Kotlin) receives the message, JSON-decodes the payload, and emits a `DeviceEventEmitter` event tagged e.g. `cablesnap.watch.setComplete`.
  - A JS-side singleton subscriber (started by the new `useWatchSync(sessionId)` hook) calls `addSet`/etc. on the JS thread.
  - **Critical:** if the JS bundle is not loaded (app cold-killed), the message is queued in a native-side bounded buffer (≤200 events, drop-oldest) and replayed on next JS init. Without this, a cold phone with the watch active loses set events.
  - Document concurrency: queue must be thread-safe; JS subscriber must dedupe on `(sessionId, exerciseId, setNumber, completedAt)` since the native queue may replay in-flight events on reconnect.
